Open `/_profiler` to browse the list of captured profiles, then click any row to open its detail view. The detail view is organized into **built-in tabs** (request, response, performance, logs, exceptions) plus **one tab per active collector**. The screenshots below come from the [example application](/docs/example-api) with every collector enabled.

This page is a **visual tour** of the panels and tabs. For the reference side â€” the UI endpoints, the debug response headers, the filter query parameters and how to export a profile â€” see [Browsing profiles](/docs/packages/nest-profiler/browsing-profiles).

## Profiles list

`/_profiler` opens on a sticky sidebar of **views**. Each entrypoint kind (HTTP, GraphQL, Commands, RabbitMQ) gets its own page under a **Profiling** group, and every global-scope collector (Config, Routes, Schemas) is a view of its own â€” each badged with its count. Only the active view is rendered; it is picked server-side from a `?view=` link, so there is no client-side routing.

Every list carries its own filter bar and is filtered independently: the universal filters (search, duration, **performance tag** â€” slow / N+1 / chatty / large payload / no rows â€” plus an **Exception** select and an **Errors** checkbox), the HTTP-status filters (status, status class) shown only on the kinds that carry an HTTP response (HTTP, GraphQL), and filters specific to that kind (e.g. HTTP **method**, RabbitMQ **delivery**/**exchange**/**handler**). A kind hides a universal filter that is redundant for it â€” the Commands list has no **Errors** checkbox, its **Status** filter already asking the same question.

## GraphQL operations

Requires [`@eleven-labs/nest-profiler-graphql`](/docs/packages/nest-profiler-graphql) - see the [GraphQL support section](/docs/getting-started#graphql-support) in Getting started.

GraphQL operations get their own **GraphQL** view in the sidebar (separate from REST requests), each row showing the operation type (**QUERY**, **MUTATION**, or **SUBSCRIPTION**) and the operation or field name. Its filter bar adds an **Operation** filter to narrow the list by operation type.

![GraphQL view listing QUERY and MUTATION operations alongside their status](/screenshots/profiler/graphql-list.png)

Opening an operation lands on its dedicated **GraphQL** detail tab, displaying:

- **Operation type** badge (Query / Mutation / Subscription)
- **Operation name** if present in the request
- **Field name** - the entry-point resolver
- **Query** - syntax-highlighted GraphQL document
- **Variables** - the variables object, syntax-highlighted as JSON
- **Response** - the `{ data, errors }` envelope returned to the client
- **Request headers** - the underlying HTTP headers

![GraphQL detail tab with operation type, name, syntax-highlighted query and variables (Query)](/screenshots/profiler/graphql-request.png)

![GraphQL detail tab for a Mutation with input variables](/screenshots/profiler/graphql-mutation.png)

GraphQL-level errors (schema validation failures, resolver errors returned in `response.body.errors`) appear in the **Exceptions** tab with an amber `GraphQLError` badge, visually distinct from NestJS runtime exceptions which use a red badge. Since GraphQL names every error `GraphQLError`, its `extensions.code` is shown alongside â€” `BAD_REQUEST` here â€” and that code is what the **Exception** filter lists and what decides whether the operation counts as an [error](/docs/packages/nest-profiler/error-classification).

![Exceptions tab showing an amber GraphQLError badge with its BAD_REQUEST code, validation message and location](/screenshots/profiler/graphql-error.png)

![Request tab showing a POST create with its method, headers and JSON request body](/screenshots/profiler/request.png)

The **Response** tab shows the status, response headers, and body returned by the handler.

The **Performance** tab groups request duration, memory, and timestamps.

![Performance tab with duration, memory, and request timeline](/screenshots/profiler/performance.png)

The **Logs** tab lists entries captured through `createProfilerLogger()`. Each row shows the message first - with structured payloads rendered as a JSON block under it - then the logger context name. See the [Log capture](/docs/packages/nest-profiler/logs) page for the supported logger conventions.

![Logs tab showing messages with context names and a structured JSON payload](/screenshots/profiler/logs.png)

The **Exceptions** tab lists what the profile captured, each with its message and stack trace â€” here the `InternalServerErrorException` behind a 500. A captured exception is not automatically an **error**, though: a `BadRequestException` answering `400` means the application did its job, so it appears here and under the **Exception** filter, but not under the **Errors** checkbox. See [What counts as an error](/docs/packages/nest-profiler/error-classification).

## Collectors

The **Timeline** collector renders a **unified trace waterfall** â€” one nested, causal view that merges every timed operation of the request on a single time axis: the lifecycle phases you add with `startSpan()`, outgoing HTTP calls, database queries, and (with [`@eleven-labs/nest-profiler-graphql`](/docs/packages/nest-profiler-graphql)) each GraphQL `resolveField`. Spans nest by causality â€” a query issued while resolving a field appears under that field â€” with collapsible subtrees, so an N+1 shows up as one field with its repeated child queries.

![Timeline waterfall of a GraphQL products query â€” an SQL root resolver, one GraphQL field span per product and its nested MongoDB reviews query, each flagged N+1](/screenshots/profiler/timeline.png)

Spans come in two families, and the distinction is what makes the numbers add up. **Containers** â€” the request root, the phases you open with `startSpan()`, and GraphQL field spans â€” group the work they triggered: their duration _overlaps_ their children's instead of adding to it, and their own cost is the **self-time** shown under the total. **Leaves** â€” HTTP calls and database statements â€” are where the time is actually spent; a `transaction` row is the one synthetic container the SQL collectors add, wrapping a `BEGIN â€¦ COMMIT/ROLLBACK` run around the statements it delimits. This is also why the `% of total` column does not sum to 100%: containers overlap their children, and concurrent spans (anything fired with `Promise.all`) overlap each other. Concurrency is never mistaken for nesting â€” two calls that merely overlap in time stay siblings; only an explicitly reported parent or a genuine container adopts a span.

Durations are captured with a sub-millisecond clock (`performance.timeOrigin + performance.now()`) and displayed in the unit that fits them â€” `400ns`, `421Âµs`, `7.12ms`, `1.23s`, up to `2m 5s` for a long-running command or consumer â€” so a value is never rounded into a misleading `0ms` nor into an unreadable pile of digits. A container that stopped being timed before the work it triggered resolved is widened onto its subtree, so a parent bar always covers what hangs under it.

Each bar shows its **self-time** (its own duration minus its children) next to the total, carries the same **performance tags** as its source entry (a repeated query is badged `N+1`, a slow one `slow`), and â€” for HTTP and database spans â€” links straight to the matching panel and highlights the exact row (the Database group even opens the right TypeORM/MongoDB sub-tab). Click a span's label to expand its full SQL/URL and metadata inline.

The waterfall makes an optimisation legible at a glance. Batching the same `products { reviews }` query with a per-request DataLoader collapses the four repeated reviews queries into a single load: the `N+1` tags and the Performance banner disappear, and the field spans that resolved from the loader's cache have no child query at all.

![Same query with DataLoader batching â€” one reviews query for all products, no N+1 tags and no Performance banner](/screenshots/profiler/timeline-dataloader.png)

Above the waterfall, a flat **Request Lifecycle** band breaks the request into its phases â€” `guards`, then `controller`, plus `validation` when [`@eleven-labs/nest-profiler-validator`](/docs/packages/nest-profiler-validator) is installed. It is a Symfony-profiler-style overview that stays out of the causal tree. `middleware` and the network send have no dedicated NestJS hook, so they are not shown; disable the band entirely with `lifecycleSpans: false`.

The **Database** collector shows SQL queries (TypeORM or MikroORM) with their SQL type and duration, plus a metadata line per query â€” rows affected/returned and the connection, rendered as a chip (`localhost:5432 / shop`). The panel header sums the rows read.

![Database tab showing a profiled SQL SELECT query](/screenshots/profiler/database.png)

A zero-row `UPDATE`/`DELETE` is a silent failure â€” the query ran but changed nothing (a mismatched `WHERE`). It is highlighted, its `0 rows` count turns amber, and the `zero-rows` tag lights the **Performance** banner and colours the Database tab:

![Database tab of a PATCH that affected 0 rows, with a "Performance â€” 1 issue detected" banner reading "No rows" and the query showing 0 rows](/screenshots/profiler/database-zero-rows.png)

The **MongoDB** collector (requires [`@eleven-labs/nest-profiler-mongoose`](/docs/packages/nest-profiler-mongoose)) shows Mongoose queries and aggregations with their operation, collection, duration, result/affected count, and connection.

![MongoDB tab showing a profiled Mongoose query with collection, operation and duration](/screenshots/profiler/mongodb.png)

The **HTTP Client** collector shows outgoing requests triggered by `HttpService`.

Each delivery becomes its own profile: a dedicated **RabbitMQ** view, and a **Message** detail tab showing the exchange, routing key, handler, delivery metadata and JSON payload.
